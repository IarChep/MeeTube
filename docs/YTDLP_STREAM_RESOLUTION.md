## 1. Общая картина: конвейер целиком

yt-dlp никогда не «скачивает страницу видео и парсит теги». Он говорит с приватным InnerTube API
(`youtubei/v1/*`) так, будто он — официальное приложение YouTube (web/android/ios/tv). Весь путь от
`videoId` до готового URL:

```
videoId
  │
  ├─(1) выбрать набор клиентов (tv, ios, web, android_vr, …)
  │
  ├─(2) для каждого клиента: POST /youtubei/v1/player
  │        context(client) + signatureTimestamp(sts) + [player poToken]
  │        └─► player response  ──►  streamingData
  │
  ├─(3) собрать форматы из streamingData:
  │        formats / adaptiveFormats  (прямые https)
  │        hlsManifestUrl             (m3u8)
  │        dashManifestUrl            (mpd)
  │
  ├─(4) для каждого URL достроить его до «скачиваемого»:
  │        а) расшифровать подпись:  signatureCipher{url,s,sp} → &sp=<sig>
  │        б) решить throttling:      n=<raw> → n=<solved>
  │        в) прицепить GVS poToken:  &pot=<tok>  (или /pot/<tok> в манифесте)
  │        (а) и (б) требуют исполнить JS из base.js — см. §5)
  │
  ├─(5) выбрать лучший формат(ы) по -f (bestvideo+bestaudio/best)
  │
  └─(6) скачать; при adaptive — склеить video+audio через ffmpeg
```

Ключевая идея: **тяжёлую крипту (sig/n) считает JavaScript из плеера YouTube (`base.js`)**, а yt-dlp
лишь находит нужные функции, исполняет их во внешнем JS-движке и **кеширует результат**, чтобы не
гонять движок на каждый URL.

Второй слой сложности — **анти-бот**: часть форматов отдаётся только при наличии `poToken` (аттестация
BotGuard) и/или только определённым клиентам. yt-dlp обходит это, **подбирая клиента**, которому URL
отдают без токена (`android_vr`, `tv`), и подключая внешнего провайдера poToken там, где без него никак.

## 2. Шаг 1 — запрос `/youtubei/v1/player`

Источник форматов — ответ эндпоинта `player`:

```
POST https://www.youtube.com/youtubei/v1/player?key=…&prettyPrint=false
Content-Type: application/json
```

Тело (упрощённо):

```jsonc
{
  "context": {
    "client": {                       // ← «кто мы»; определяет, какие форматы отдадут
      "clientName": "ANDROID_VR",
      "clientVersion": "1.60.19",
      "androidSdkVersion": 32,
      "userAgent": "com.google.android.apps.youtube.vr.oculus/1.60.19 …",
      "hl": "en", "gl": "US"
    }
  },
  "videoId": "dQw4w9WgXcQ",
  "playbackContext": {
    "contentPlaybackContext": {
      "signatureTimestamp": 20256      // ← sts, извлекается из base.js (см. ниже)
    }
  },
  "serviceIntegrityDimensions": {
    "poToken": "<player-poToken>"      // ← только для android/ios (см. §6)
  }
}
```

Три вещи, которые yt-dlp кладёт в этот запрос и без которых потоки не приедут корректными:

- **`context.client`** — «легенда». Именно от клиента зависит, придут ли ссылки зашифрованными,
  придут ли вообще прямые URL (а не SABR-only), и потребуется ли poToken. Разные клиенты запрашиваются
  ради разных наборов форматов (§7).
- **`signatureTimestamp` (`sts`)** — целое число, «версия» плеера. yt-dlp вытаскивает его из `base.js`
  (`sts` / `signatureTimestamp` в коде плеера). Если `sts` не совпадает с версией `base.js`, из которой
  взяты функции расшифровки, подписи будут решены неправильно → HTTP 403 при скачивании.
- **`poToken`** (player-scope) — аттестация BotGuard; кладётся в `serviceIntegrityDimensions` только для
  клиентов, которым он реально нужен (`android`, `ios`). Без него эти клиенты отдают
  `LOGIN_REQUIRED`/усечённые данные.

`base.js` — это `https://www.youtube.com/s/player/<hash>/player_ias.vflset/…/base.js`. Из него yt-dlp
берёт: (1) `sts`; (2) функцию расшифровки подписи; (3) функцию `n`-трансформа. URL плеера yt-dlp
находит из webpage или из известного «iframe API» пути. Один `base.js` обслуживает все видео сессии,
поэтому функции решаются один раз и кешируются по `player_url`.

**Проверка статуса.** В ответе есть `playabilityStatus.status`: `OK` / `LOGIN_REQUIRED` /
`UNPLAYABLE` / `AGE_VERIFICATION_REQUIRED` / `CONTENT_CHECK_REQUIRED` и т.п. Если не `OK`, yt-dlp
пробует другого клиента из списка (fallback). Именно поэтому запрашивается **несколько** клиентов —
их ответы дополняют друг друга и подстраховывают.

## 3. Шаг 2 — `streamingData`: три формы потоков

Всё скачиваемое лежит в `player_response.streamingData`. Есть три принципиально разных источника:

| Поле | Что это | Итог |
|------|---------|------|
| `formats` | **progressive** — видео+аудио в одном файле (muxed). Почти всегда только itag **18** (360p mp4), исторически бывал 22 (720p, сейчас редко/нет) | одна прямая ссылка, ffmpeg не нужен |
| `adaptiveFormats` | **adaptive** — раздельные дорожки: video-only и audio-only (как DASH-репрезентации) | нужно взять лучшее видео + лучшее аудио и склеить |
| `hlsManifestUrl` | ссылка на **HLS** плейлист `.m3u8` (мастер + медиаплейлисты) | сегментное скачивание |
| `dashManifestUrl` | ссылка на **DASH** манифест `.mpd` | сегментное скачивание |

Каждый элемент `formats`/`adaptiveFormats` — объект с полями:

```jsonc
{
  "itag": 251,
  "mimeType": "audio/webm; codecs=\"opus\"",
  "bitrate": 145509, "averageBitrate": 129648,
  "width": 1920, "height": 1080, "fps": 30,         // для видео
  "qualityLabel": "1080p", "quality": "hd1080",
  "audioQuality": "AUDIO_QUALITY_MEDIUM",
  "audioSampleRate": "48000", "audioChannels": 2,
  "contentLength": "3200145", "approxDurationMs": "212041",
  "initRange": {"start":"0","end":"265"},           // для DASH-нарезки
  "indexRange": {"start":"266","end":"1234"},
  "lastModified": "…",

  // взаимоисключающе:
  "url": "https://rr3---sn-…​.googlevideo.com/videoplayback?…",   // либо готовый URL
  "signatureCipher": "url=https%3A%2F%2F…&s=AOq0QJ…&sp=sig"        // либо зашифрованный
}
```

Три ключевых наблюдения:

1. **`url` vs `signatureCipher`** — формат приходит либо с прямым `url`, либо с полем
   `signatureCipher` (в некоторых клиентах — `cipher`), которое надо расшифровать (§4а).
2. **`n` есть почти всегда** — даже прямой `url` несёт query-параметр `n=<value>`, который надо
   пересчитать, иначе googlevideo режет скорость/отдаёт 403 (§4б).
3. **Формат может прийти БЕЗ `url` и без `signatureCipher`** — это **SABR-only** формат. yt-dlp его
   пропускает с предупреждением (§10).

## 4. Шаг 3 — расшифровка URL: `sig`, `n`, `pot`

Это ядро «расшифровки». Готовый скачиваемый googlevideo-URL = исходный URL + три доработки.
yt-dlp делает это в **два прохода**: сначала собирает все sig/n-«челленджи» со всех форматов,
**решает их пачкой один раз**, потом достраивает URL — так JS-движок дёргается минимально.

### 4а. Подпись (`signatureCipher` → `sig`)

Поле `signatureCipher` — это urlencoded-строка `url=…&s=…&sp=…`:

- `url` — базовый googlevideo URL (без подписи);
- `s` — **зашифрованная подпись** (скрэмблированная строка, ~150–260 символов);
- `sp` — **имя параметра**, под которым подпись надо прицепить к URL (обычно `sig`, иногда
  `signature`).

Функция расшифровки живёт в `base.js`. По сути это цепочка простых операций над массивом символов:

```js
// упрощённый вид декодера подписи из base.js
function decodeSig(a){
  a = a.split("");
  Xx.reverse(a);          // reverse всего массива
  Xx.splice(a, 3);        // splice(a, b) → a.splice(0, b) — отрезать первые b
  Xx.swap(a, 27);         // swap(a, b) → поменять местами a[0] и a[b % len]
  Xx.reverse(a);
  // …ещё несколько таких же…
  return a.join("");
}
```

Важное свойство: все операции (`reverse`, `splice`, `swap`) — это **чистая перестановка индексов**.
Поэтому классическая оптимизация yt-dlp: прогнать декодер на «эталонной» строке `[0,1,2,…,L-1]`,
получить перестановку `spec` для данной длины `L`, и дальше применять её **без JS**:

```python
sig = ''.join(s[i] for i in spec)          # spec кешируется по (player_url, len(s))
```

Итог прицепляется к URL: `…&<sp>=<sig>`, например `…&sig=A8f2…`.
(В современном yt-dlp сам декодер прогоняется через внешний JS-движок EJS — §5 — но результат для
данной длины по-прежнему кешируется, а перестановочная модель остаётся точным описанием того, *что*
делает функция.)

### 4б. Throttling-параметр `n`

Каждый googlevideo-URL несёт `&n=<value>`. Если оставить `n` как есть, YouTube **троттлит** скачивание
(десятки КБ/с) или отдаёт 403. Нужно заменить `n` на результат `n`-трансформа из `base.js`.

В отличие от подписи, `n`-функция — **настоящее вычисление**, а не перестановка: длинный
обфусцированный код с массивами, битовыми операциями, подстановочными таблицами, циклами. Свести к
перестановке нельзя — надо реально исполнить JS.

- Решается **по одному разу на каждое различное значение `n`** и кешируется по значению.
- Прогрессивный URL: параметр запроса — `&n=<solved>`.
- Манифесты (HLS/DASH): `n` часто закодирован **сегментом пути** — `…/n/<solved>/…` (googlevideo
  умеет и path-стиль параметров), и yt-dlp правит его прямо в URL сегментов.

Если `n` не пересчитан — это как раз типичная причина «скачивается по 40 КБ/с» или «через минуту 403».

### 4в. GVS poToken

Третья добавка — **googlevideo poToken** (GVS-scope, отличается от player-poToken из §2). Прицепляется
к финальному URL там, где политика клиента этого требует (§6):

- прогрессив/DASH-URL: `&pot=<token>`;
- HLS/DASH-манифест: сегмент пути `…/pot/<token>/…`.

yt-dlp сам токен **не генерирует** — берёт у внешнего провайдера (§6).

### Порядок сборки одного URL

```
base_url (из signatureCipher.url или из format.url)
   → + &<sp>=<decodeSig(s)>          (если был signatureCipher)
   → заменить n=<raw> на n=<nsig(raw)>
   → + &pot=<gvs_token>              (если политика клиента требует)
   → готовый googlevideo URL
```

## 5. Слой исполнения JavaScript (jsc / EJS)

Чтобы решить §4а и §4б, нужно исполнить куски `base.js`. Здесь у yt-dlp за последний год произошёл
большой сдвиг.

### Было: Python-интерпретатор JS (`jsinterp.py`)

Исторически yt-dlp нёс **собственный интерпретатор JavaScript на Python** (`yt_dlp/jsinterp.py`,
`JSInterpreter`): парсил `base.js` регэкспами, находил функции подписи/`n` по имени и «исполнял» их
на Python. Файл `jsinterp.py` **всё ещё есть в репозитории** (используется как библиотека/фоллбэк для
других экстракторов), но для YouTube sig/n он больше **не** является основным путём: обфускация плеера
стала слишком агрессивной для регэксп-поиска и Python-исполнения.

### Стало: EJS + внешний JS-движок (`extractor/youtube/jsc/`)

Современный путь — **JSC (JS Challenge) фреймворк**: `yt_dlp/extractor/youtube/jsc/`.

```
jsc/
├── provider.py        ← публичный API: JsChallengeProvider, JsChallengeType (N, SIG)
├── _director.py       ← выбирает и вызывает провайдера, bulk-solve
├── _registry.py       ← реестр провайдеров + предпочтения (preference)
├── README.md
└── _builtin/
    ├── ejs.py         ← базовый провайдер: собирает EJS-солвер, гонит его в рантайме
    ├── deno.py        ← рантайм: Deno
    ├── node.py        ← рантайм: Node.js
    ├── bun.py         ← рантайм: Bun
    ├── quickjs.py     ← рантайм: QuickJS / quickjs-ng
    └── vendor/        ← вендоренный JS-код солвера (EJS)
```

**EJS** (`yt-dlp/ejs` — отдельный проект, вендоренный сюда) — это не «ещё один рантайм», а **сам
солвер на JavaScript**, который:

1. **парсит `base.js` в AST** (парсер `meriyah`) — не регэкспами;
2. находит функции подписи и `n` **по структуре кода** (`matchesStructure`), а не по имени — устойчиво
   к переименованиям/обфускации;
3. **сериализует** нужную функцию обратно в JS (`astring`) и исполняет её на входных данных;
4. отдаёт результат как JSON в stdout.

Базовый класс `ejs.py` строит этот солвер-бандл и передаёт код в абстрактный `_run_js_runtime()`,
который реализуют конкретные провайдеры-рантаймы. Схема запуска тривиальна: собрать
`solver.js + input`, запустить `deno|node|bun|quickjs`, прочитать stdout.

Откуда берётся сам солвер-скрипт (в порядке приоритета): установленный **PyPI-пакет** →
локальный **кеш** → **встроенная** копия (`_builtin/vendor/`) → **скачать из web** (репозиторий
`yt-dlp/ejs`). Есть dev-опции (`dev`, `repo`, `script_version`) для отладки/оверрайда.

**Рантайм** нужен внешний — один из `deno` / `node` / `bun` / `quickjs`. `quickjs-ng` рекомендуется
не ниже `0.12.0`, иначе очень медленно. Провайдер выбирается по доступности и `preference`;
пользователь может форсировать через `--extractor-args`.

### Кеширование

- **sig**: перестановка `spec` по `(player_url, длина s)` (диск-кеш `sigfuncs`) — JS дёргается один раз
  на длину.
- **`n`**: результат по значению `n` (в рамках сессии/кеша) — JS дёргается один раз на уникальное
  значение.
- Оба на общий player_url; поэтому для плейлиста из сотен видео движок реально запускается считанные
  разы.

> Практический вывод для порта (например, MeeTube): наивный «регэкспом достать функцию и повторить»
> для `n` **не работает** на современном обфусцированном плеере — нужен именно AST-солвер в реальном
> JS-движке (`quickjs-ng`). Подпись (перестановка) переносится проще, но её функцию всё равно надёжнее
> исполнить, чем угадать регэкспом.

## 6. poToken: две разные привязки

**poToken** («Proof of Origin Token») — токен аттестации, который YouTube выпускает через **BotGuard**
(тот же движок, что и reCAPTCHA-подобные challenge). Он доказывает «запрос пришёл из настоящего
клиента». Есть **два разных токена с разной областью действия** — их постоянно путают:

| Токен | Куда кладётся | Кому нужен |
|-------|---------------|------------|
| **Player poToken** | в тело `/player`: `serviceIntegrityDimensions.poToken` | `android`, `ios` |
| **GVS poToken** | на URL потока/манифеста: `&pot=` или `/pot/` | web-семейство, `android`, `ios` (HLS часто опционально) |

**yt-dlp сам токены не генерирует.** BotGuard требует запускать GoogleTag/бинарник-челлендж. yt-dlp
делегирует это внешнему провайдеру через `fetch_po_token_func`:

- **`bgutil-ytdlp-pot-provider`** — самый распространённый; поднимает HTTP-сервис (по умолчанию
  `:4416`), который выпускает токены (на базе `visitorData`);
- либо script-провайдер (локальный бинарь).

Оба токена завязаны на **`visitorData`** (идентификатор анонимной сессии из InnerTube-ответа), поэтому
`visitorData` для минта токена и для запросов должен быть один и тот же.

### Политики: когда токен вообще нужен

Ключ к обходу — **poToken требуется не всегда**. В `_base.py` для каждого клиента и каждого протокола
(HTTPS / DASH / HLS) задана политика `GVS_PO_TOKEN_POLICY` с флагами (грубо): `required` /
`recommended` / `not_required`.

- Формат **пропускается** (`MISSING POT`) только если политика `required` **и** токена нет.
- `recommended` — yt-dlp попробует достать токен, но формат оставит в любом случае.
- Исключения (токен **не** нужен):
  - **itag 18** — жёстко освобождён (exempt);
  - **HLS** во всём web-семействе — `required=False`;
  - **всё** на `android_vr` / `web_embedded` / `tv` — у них нет ключа политики → по умолчанию всё
    «не требуется»;
  - `android` / `ios`, если уже был отправлен **player-token** (`not_required_with_player_token`);
  - premium-подписка на web HTTPS/DASH (`not_required_for_premium`).

По сути GVS-poToken реально обязателен в основном для **web-семейства, HTTPS/DASH, анонимно**
(и не itag 18). Отсюда и стратегия §7.

## 7. Спуфинг клиентов: как обходят стену

Список клиентов — `INNERTUBE_CLIENTS` в `_base.py`. Каждый несёт свои `clientName`/`clientVersion`,
`userAgent`, требования к авторизации и — главное — **свою политику форматов и poToken**. yt-dlp
запрашивает `/player` **несколькими** клиентами и объединяет их `streamingData`.

Зачем несколько клиентов:

- **`web` / `web_safari`** — дают богатый набор adaptive-форматов (в т.ч. VP9/AV1, высокие разрешения),
  но именно им чаще всего нужен **GVS poToken** и они первыми упираются в **SABR** (§10).
- **`android_vr`** — «золотой» для анонима: по умолчанию **poToken не требуется** (нет ключа политики),
  часто отдаёт прямые прогрессив/adaptive URL. Ходовой обходной клиент.
- **`tv` / `tv_embedded` (TVHTML5)** — устойчив, часто нужен для авторизованных сессий и обхода
  возрастных ограничений; poToken по умолчанию не требуется. Ответы имеют «плиточную» (tile) форму.
- **`ios`** — исторически отдавал HLS без `visitorData`-минта; удобен как fallback для HLS.
- **`android`** — требует player-poToken; при его наличии GVS-токен для потоков уже не нужен.
- **`mweb`, `web_embedded`, `web_creator`, `web_music`, `*_music`, `*_creator`** — нишевые (музыка,
  встраивание, авторские инструменты, обход отдельных ограничений).

Управление: `--extractor-args "youtube:player_client=android_vr,web_safari,tv"`. По умолчанию yt-dlp
берёт небольшой набор (обычно из `tv` + web-семейства + `ios`), а для авторизованных сессий смещается
в сторону `tv`/`web`. Точный дефолт меняется от релиза к релизу вслед за анти-бот-политикой YouTube.

Стратегия в одном предложении: **выбрать клиента, которому нужный формат отдают с прямым URL и без
обязательного poToken; если такого нет — подключить провайдера poToken.**

## 8. Выбор формата и лучшего качества потока

Собрав и достроив URL всех форматов, yt-dlp выбирает, что скачать. За это отвечает селектор `-f`.

### Прогрессив vs adaptive

- **Прогрессив** (`formats`, itag 18): один файл, но потолок 360p — «лучшее качество» так не получить.
- **Adaptive** (`adaptiveFormats`): раздельные video-only и audio-only. Максимум качества берётся
  **только** отсюда: лучшее видео + лучшее аудио, затем склейка `ffmpeg`.

### Дефолт `-f bestvideo*+bestaudio/best`

Логика по умолчанию:

1. взять **лучшую видео-дорожку** (`bestvideo`) и **лучшую аудио-дорожку** (`bestaudio`), склеить;
2. если раздельных нет — взять лучший **muxed** (`best`).

«Лучшее» ранжируется по (в порядке важности): разрешение (height) → fps → битрейт → предпочтение
кодека → размер. Тай-брейки настраиваются через `-S/--format-sort`.

### Кодеки и «качество картинки» видео

Один и тот же 1080p приходит в разных кодеках, и это влияет и на качество, и на совместимость:

- **Видео**: `av01` (AV1, лучший на бит, тяжёлый в декоде) → `vp9`/`vp09` → `avc1` (H.264, самый
  совместимый). itag'и группируют это: например `137/248/303/271/…` — 1080p в разных кодеках,
  `313/315/…` — 2160p, `394…399` — AV1.
- **Аудио**: `opus` (webm, itag 251/250/249) vs `aac`/`mp4a` (m4a, itag 140/139), плюс itag'и Ambisonic/
  многоканальные и языковые аудиодорожки.
- **HDR / высокий fps / VR** маркируются в `qualityLabel`/кодеке; `-S` умеет предпочитать/избегать
  (`-S "codec:vp9"`, `+res`, `hdr` и т.п.).

По умолчанию yt-dlp для одинакового разрешения предпочитает более эффективный/совместимый кодек по
встроенной таблице приоритетов; при `--format-sort` пользователь переопределяет (например, ради
H.264 для железа без AV1-декодера — как раз кейс Nokia N9).

### Языковые дорожки

Если у видео несколько аудиоязыков, у аудио-форматов появляется `language`/`audioTrack`. yt-dlp умеет
выбирать нужный (`-S "lang:en"` / фильтры формата).

## 9. HLS и DASH манифесты

Кроме прямых форматов, `streamingData` может дать `hlsManifestUrl` (m3u8) и `dashManifestUrl` (mpd):

- **HLS**: мастер-плейлист со списком качеств → медиаплейлисты с сегментами. Часто это единственный
  путь для некоторых клиентов (например `ios`) и для лайв-стримов.
- **DASH**: `.mpd` с `Representation` на каждое качество; сегменты берутся по `initRange`/`indexRange`
  либо из SegmentTemplate.

Важное: URL **внутри** манифестов — те же googlevideo-ссылки, и их тоже надо доработать. В манифестах
`n` и `pot` обычно закодированы **сегментами пути** (`/n/<solved>/`, `/pot/<token>/`), а не query-
параметрами — yt-dlp правит их при разборе манифеста. Скачивание идёт по-сегментно (можно параллелить),
затем сборка.

Для **лайв-стримов** и premiere форматы часто только манифестные; прогрессив-URL там нет.

## 10. Стена SABR / UMP

Самое серьёзное ограничение последних лет.

**SABR** (Server-side Adaptive BitRate) — новый транспорт YouTube: клиент не забирает готовые сегменты
по обычным URL, а **POST'ит на `/videoplayback`** тело в формате **UMP** (Universal Media Playback —
protobuf-поток), а сервер сам решает, какие куски какого качества прислать (ABR на стороне сервера).
Формат при этом приходит в `streamingData` **без `url` и без `signatureCipher`** — только через
`serverAbrStreamingUrl` + SABR-конфиг.

**yt-dlp SABR/UMP не реализует** (в кодовой базе нет ни `sabr`, ни `serverAbr`, ни `UMP`, ни разбора
`videoplayback`-протокола). Поведение:

- SABR-only форматы (без URL) — **пропускаются** с предупреждением вроде «forcing SABR / no formats».
- Анти-SABR-стратегия = **выбрать клиента, который всё ещё отдаёт настоящие URL** (тот же `android_vr`,
  `tv`, иногда `ios`-HLS) **и при необходимости дать poToken**.

Сам SABR/UMP как протокол реализован в стороннем проекте **LuanRT `googlevideo`**, а не в yt-dlp. Порт,
которому нужен доступ к SABR-only форматам, вынужден реализовывать UMP сам.

## 11. Лучшее качество картинки: thumbnail и storyboard

«Лучшее качество изображения» = превью (thumbnail) видео. Два источника, yt-dlp объединяет оба и
выбирает по разрешению.

### Источник 1 — предсказуемые URL `i.ytimg.com`

Превью лежат по шаблону:

```
https://i.ytimg.com/vi/<videoId>/<name>.jpg          ← JPEG
https://i.ytimg.com/vi_webp/<videoId>/<name>.webp    ← WebP (меньше вес, лучше качество)
```

где `<name>` по возрастанию качества:

| `<name>` | Разрешение (типично) |
|----------|----------------------|
| `default` | 120×90 |
| `mqdefault` | 320×180 |
| `hqdefault` | 480×360 |
| `sddefault` | 640×480 |
| `hq720` | 1280×720 |
| `maxresdefault` | 1920×1080 (или исходное) |

`maxresdefault` — самое большое, но **существует не для всех видео** (зависит от того, в каком
разрешении был загружен ролик). yt-dlp формирует список кандидатов и проверяет доступность (пустой/404
превью отбрасывается), выбирая наибольшее реально существующее. Кадры `1.jpg`/`2.jpg`/`3.jpg` — это
превью-раскадровка (три момента ролика).

### Источник 2 — `thumbnails` из player/next response

В ответе InnerTube есть массив `videoDetails.thumbnail.thumbnails[]` (и в renderer'ах) с уже готовым
списком `{url, width, height}`. yt-dlp добавляет их к кандидатам.

### Выбор и запись

- Внутри yt-dlp каждому thumbnail присваивается `preference`/`id` по разрешению; при `--write-thumbnail`
  берётся наибольшее. `--list-thumbnails` показывает все варианты.
- WebP-варианты (`vi_webp`) предпочтительнее по размеру/качеству, если поддерживаются.

### Storyboard (превью при перемотке)

Отдельно от статичного превью есть **storyboard** — спрайт-атласы кадров для полоски перемотки. Их
спецификация лежит в `playerResponse.storyboards.playerStoryboardSpecRenderer.spec` (шаблон URL с
`$M/$N` плейсхолдерами и уровнями качества `L0/L1/L2/L3`). yt-dlp умеет их разворачивать в отдельный
«storyboard»-формат. Для обычного скачивания превью это не нужно, но это тоже часть «картиночного»
конвейера.

## 12. Субтитры (`timedtext`)

Не поток и не картинка, но часть того же URL-конвейера и важный подводный камень при портировании.

- Список дорожек — `playerResponse.captions.playerCaptionsTracklistRenderer.captionTracks[]`
  (человеческие) и автогенерённые. Форматы: `srv1/srv2/srv3/ttml/vtt/json3`.
- URL субтитров — тот же `timedtext`-эндпоинт c уже вписанными подписанными параметрами.
- **Подводный камень (актуален для MeeTube):** серверные `timedtext`/`googlevideo` URL уже содержат
  `%`-escape'ы (`%3D`, `%2C`). Если такой URL пропустить через `QUrl(QString)`, Qt **двойным
  кодированием** превратит `%3D`→`%253D` и подпись сломается → 403. Их надо заносить в Qt как
  `QUrl::fromEncoded(url.toUtf8())`. (В yt-dlp/Python этой проблемы нет — она специфична для Qt.)

## 13. Сквозной пример

Скачать `dQw4w9WgXcQ` в максимальном качестве, анонимно:

```
1. base.js:  найти player_url → извлечь sts, декодер подписи, n-функцию.
             (EJS распарсит base.js в AST, найдёт функции по структуре.)

2. /player:  POST context(ANDROID_VR) + signatureTimestamp=sts
             ← playabilityStatus.status == OK
             ← streamingData.adaptiveFormats = [ itag137(1080p avc), itag248(1080p vp9),
                                                  itag251(opus), itag140(aac), … ]

3. форматы:  собрать sig-челленджи (s каждого зашифрованного формата)
             + n-челленджи (n каждого URL)  →  bulk-solve в quickjs-ng один раз
             для каждого формата:  url + &sig=<perm(s)> ; заменить n ; (pot не нужен — android_vr)

4. выбор:    -f bestvideo+bestaudio  →  itag248 (1080p vp9)  +  itag251 (opus)

5. качать:   скачать обе дорожки  →  ffmpeg -c copy склеить в .mkv/.webm

6. превью:   кандидаты maxresdefault.webp → hq720 → sddefault …  проверить, взять наибольшее живое.
```

Если бы `android_vr` вернул только SABR-only форматы — yt-dlp пропустил бы их и перешёл к следующему
клиенту (`tv`/`web`+poToken); если бы понадобился web-HTTPS-формат анонимно — подтянул бы GVS-poToken у
`bgutil` провайдера.

## 14. Что из этого важно для MeeTube

Проекция алгоритма на playback-путь MeeTube (`src/core/media/`, InnerTube-движок):

- **`n`-параметр — вероятный корень прогрессив-403 на itag 18.** yt-dlp тянет `android_vr`/itag18
  вообще без poToken, и это работает, потому что он **решает `n`** для всех клиентов через JS. У
  MeeTube JS-движка нет → `n` не решён → троттлинг/403. Проверять: несёт ли наш `android_vr`/itag18 URL
  `&n=`; если да — портировать `n`-трансформ (нужен реальный AST-солвер в `quickjs-ng`, регэксп не
  спасёт), а не BotGuard.
- **poToken реально нужен реже, чем кажется.** itag 18, `android_vr`, `tv`, HLS web — без него.
  Смотри политики §6, прежде чем строить дорогой BotGuard-минтер.
- **Максимальное качество — только adaptive + склейка.** Прогрессив упирается в 360p. Для N9 разумно
  предпочитать `avc1`+`aac` (аппаратный декод), а не `av01`/`vp9`/`opus`.
- **SABR — самостоятельная большая работа** (UMP/protobuf, порт из LuanRT `googlevideo`); стратегия
  «избегать SABR выбором клиента» дешевле.
- **`QUrl::fromEncoded`** для всех серверных URL (googlevideo, timedtext) — иначе двойное
  `%`-кодирование ломает подпись (§12).

Смежные заметки проекта: [`youtube-potoken-wall`], [`ytdlp-stream-resolution`],
[`docs/superpowers/specs/2026-07-09-youtube-stream-player-design.md`](superpowers/specs/2026-07-09-youtube-stream-player-design.md).
